# 🐞 노래 신청 명령어 파싱 및 대기열 누락 현상 분석 보고서

## 1. 현상 요약
- **발생 상황**: 사용자가 1,000 치즈 후원과 함께 `"유저가 입력한 후원메시지입니다"`라는 메시지를 전송함.
- **기대 결과**: 노래책에 있는 `"입력한"`(비용: 1000 치즈) 곡이 대기열에 추가되고, 1000 치즈가 차감되어야 함.
- **실제 결과**: 사용자가 기대한 곡이 대기열에 추가되지 않고 차감도 일어나지 않음 (지갑에 1000치즈 누적).

---

## 2. 원인 추적 (Root Cause Analysis)

문제의 근본적인 원인은 결제나 트랜잭션 오류가 아닌, **'명령어 인자 추출 방식'**과 **'미등록 곡의 비용 처리 정책(FREE)'**이 겹치면서 발생한 논리적 예외 케이스(Edge Case)입니다. 시스템은 오류 없이 완벽하게 작동했으나 사용자의 의도와 다르게 해석되었습니다.

### 🔍 단계별 실행 흐름 분석

1. **명령어 매칭 (정상)**
   - 수신된 전체 메시지: `"유저가 입력한 후원메시지입니다"`
   - `UnifiedCommandHandler`는 메시지가 `"유저가"`로 시작함을 감지하고 정상적으로 명령어 처리를 시작합니다.

2. **곡명(Arguments) 파싱 (원인 발생 1)**
   - 키워드(`"유저가"`)를 제외한 나머지 문자열 전체를 신청할 곡명으로 파싱합니다.
   - 추출된 곡명(`request.SongTitle`): **`"입력한 후원메시지입니다"`**

3. **노래책 검색 실패 (원인 발생 2)**
   - `AddSongRequestCommand`가 DB(`SearchPersonalSongBookAsync`)에서 파싱된 곡명을 검색합니다.
   - 노래책에 등록된 실제 곡명은 `"입력한"`이므로, 검색어 `"입력한 후원메시지입니다"`와 일치하지 않아 검색에 실패합니다. (`matchedSong = null`)

4. **비용 검증 우회 (결정적 원인)**
   - 검색에 실패할 경우, 시스템은 이를 **"노래책에 없는 자유 신청곡(미등록 곡)"**으로 간주합니다.
   - 미등록 곡은 해당 명령어 자체의 기본 요금(`request.DefaultCost`)을 따르도록 설계되어 있습니다.
   - 설정 스크린샷에 따르면 `"유저가"` 명령어의 기본 비용은 **`FREE (0 치즈)`**로 설정되어 있습니다.
   - 따라서 필요 차감액(`requiredPoints`)이 0이 되어, **1000 치즈 결제 검증 로직을 무사히 통과(우회)해버립니다.** 
   - *(이때 후원하신 1000 치즈는 명령어 차감 없이 지갑에 단순 충전된 상태로 남게 됩니다.)*

5. **대기열 등록 상태 (고스트 데이터)**
   - 로직상 오류나 비용 부족이 없었기 때문에, DB(`FuncSongQueues`)와 인메모리 버퍼에는 **`"입력한 후원메시지입니다"`**라는 제목으로 곡이 정상 등록(무료)되었습니다.
   - 하지만 이 곡은 `SongBookId`가 매핑되지 않은 단순 텍스트 곡이 되었고, 사용자가 기대했던 `"입력한"` 곡이 아니었기 때문에 화면 상 "추가되지 않았다"고 인지하게 된 것입니다.

## 3. 요약 및 향후 개선 방향 (조언)

현재 개편하신 결제 아키텍처(정책 B) 자체의 결함이 아니라, **"후원 메시지가 곡 제목에 섞여 들어가는 현상"** 때문입니다. 코드는 설계된 논리대로 완벽히 작동했으나, 유저의 자유로운 입력 패턴이 예측 범위를 벗어난 케이스입니다.

### 💡 권장 해결책 (조언)
1. **명령어 엄격 매칭 (Strict Match) 모드 도입**: 
   - 노래 신청 명령어 옵션에 "노래책에 등록된 곡만 신청 허용"을 추가하여, 검색된 곡이 없을 경우 `FREE` 곡으로 통과시키는 대신 `"곡을 찾을 수 없습니다."`라며 환불(Failure) 처리하도록 방어 로직 추가.
2. **명령어 요금 설정 확인**:
   - 노래 신청 명령어(`"유저가"`)의 기본 비용을 `FREE`가 아닌, 최소 신청 비용(예: 1000 치즈)으로 맞추어 의도치 않은 무료 등록을 방지.
3. **AI 곡명 추출 고도화**:
   - LLM을 활용해 `"입력한 후원메시지입니다"`에서 곡명인 `"입력한"`만 똑똑하게 추출해내는 프롬프트 파이프라인 추가.
