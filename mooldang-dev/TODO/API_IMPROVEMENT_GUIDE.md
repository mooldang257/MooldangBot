# 🧭 UI ↔ 백엔드 통신 규약 개선 상세 가이드 (P0~P3)

이 문서는 물댕댕봇(MooldangBot) 함대의 UI와 백엔드 간 통신 효율성을 극대화하기 위한 단계별 개선 로드맵의 상세 설명서입니다. 1인 개발 환경에서 유지보수 비용을 최소화하고 시스템의 견고함을 높이는 데 목적이 있습니다.

---

## 🔴 P0: 즉시 개선 (안정성 및 표준화)
> **최우선 순위**: 현재 발생 중인 잠재적 장애 요인을 제거하고 최소한의 형식을 맞춥니다.

### 1. 엔티티 직접 반환 제거 (DTO 전환)
- **정의**: API 응답에서 `Roulette`, `SongQueue` 같은 DB 엔티티 대신 별도로 정의된 `ResponseDto`를 반환합니다.
- **필요성**: EF Core 엔티티는 순환 참조(Circular Reference) 구조가 많아 JSON 직렬화 시 500 에러를 유발합니다. 현재의 수동 `null` 처리 방식은 누락 시 장애로 이어집니다.
- **기대 효과**: 직렬화 에러 완전 차단 및 민감한 DB 필드(비밀번호, 내부 토큰 등)의 노출 방지.

### 2. 라우트 네이밍 정규화 (kebab-case)
- **정의**: `api/SongRequest` (PascalCase)를 `api/song-request` (kebab-case)로 변경합니다.
- **필요성**: RESTful API의 글로벌 표준은 소문자 케밥 케이스입니다. 일관성 없는 네이밍은 프론트엔드 개발 시 URL을 매번 백엔드 소스에서 찾아야 하는 비효율을 초래합니다.
- **기대 효과**: 예측 가능한 API 경로 설계로 개발 생산성 향상.

### 3. URL 내 불필요한 동사 제거 (RESTful Method)
- **정의**: `/api/song/add` → `POST /api/song`으로 전환.
- **필요성**: HTTP 메서드(POST, DELETE 등)는 이미 의도를 포함하고 있습니다. URL에 동사를 중복 사용하는 것은 비표준적이며 경로가 불필요하게 길어집니다.
- **기대 효과**: 표준에 부합하는 깔끔한 API 인터페이스 구축.

## 🟠 P1: 단기 개선 (일관성 및 단순화)
> **중점**: 개발 패턴을 하나로 통일하여 새로운 기능을 추가할 때의 고민 시간을 줄입니다.

### 1. 페이지네이션 단일화 (커서 기반)
- **정의**: `offset/limit` 방식을 제거하고 모든 리스트 조회를 `CursorPagedResponse`로 통일합니다.
- **필요성**: 현재 3가지 방식이 혼재되어 리스트 기능 구현 시마다 설계 방식이 달라집니다. 커서 방식은 대량 데이터 처리에 성능 우위가 있습니다.
- **기대 효과**: 목록 처리 로직의 재사용성(Re-usability) 극대화.

### 2. 응답 타입 규격화 (Explicit DTOs)
- **정의**: `Result<object>` 대신 `Result<ChatSettingsResponse>`와 같이 명시적 타입을 사용합니다.
- **필요성**: 익명 타입(object)은 프론트엔드에서 데이터 구조를 알 수 없어 런타임 에러(`undefined` 참조 등)를 유발하기 쉽습니다.
- **기대 효과**: 백엔드 수정 시 프론트엔드 코드의 안정성 확보.

### 3. 라우트 구조 정규화 (`api/{도메인}/{uid}/...`)
- **정의**: 모든 도메인 API에서 스트리머 식별자(uid)의 위치를 고정합니다.
- **필요성**: UID 위치가 제각각이면 공통 권한 체크 미들웨어를 적용하기 불가능합니다.
- **기대 효과**: 공통 보안 로직 적용의 기틀 마련.

### 4. API 버전 관리 라이브러리 제거
- **정의**: `Asp.Versioning` 의존성을 제거하고 단순한 경로 체계로 회귀합니다.
- **필요성**: 1인 개발 환경에서 API 버전 분리는 오버헤드입니다. 현재 일부만 적용된 상태라 오히려 복잡도만 높이고 있습니다.
- **기대 효과**: 프로젝트 복잡도 감소 및 빌드 속도 개선.

## 🟡 P2: 중기 개선 (품질 및 유지보수성)
> **중점**: 코드가 커져도 관리가 가능하도록 구조적 안전장치를 마련합니다.

### 1. Strongly-typed SignalR Hub 도입
- **정의**: `Hub` 대신 `Hub<IOverlayClient>` 인터페이스를 사용합니다.
- **필요성**: 문자열(`SendAsync("ReceiveData", ...)`) 기반 호출은 오타에 취약합니다. 인터페이스를 쓰면 컴파일 타임에 모든 에러를 잡을 수 있습니다.
- **기대 효과**: 실시간 통신 버그 제로화 및 이벤트명 일관성 유지.

### 2. 프론트엔드 API Client 통일 (handleFetch)
- **정의**: Admin 프로젝트의 SSR 처리 방식을 Studio의 `handleFetch` 방식으로 일원화합니다.
- **필요성**: 현재 프로젝트마다 서로 다른 API 연동 코드를 유지하고 있어 Docker/Nginx 설정 변경 시 대응이 누락될 위험이 있습니다.
- **기대 효과**: 프로젝트 간 통신 계층의 일관성 및 네트워크 설정 관리 용이성.

### 3. AuthController 책임 분리 (Decomposition)
- **정의**: 비대한 `AuthController`를 로그인(Auth), 신원(Identity), 우회(Proxy)용으로 분할합니다.
- **필요성**: 단일 책임 원칙(SRP) 위배로 인해 코드가 읽기 어렵고 수정 시 사이드 이펙트 파악이 어렵습니다.
- **기대 효과**: 코드 가독성 향상 및 단위 테스트 효율성 증대.

---

## 🟢 P3: 장기 개선 (자동화 및 보안 고도화)
> **중점**: 사람이 실수할 여지를 기계가 대신 검증하도록 자동화합니다.

### 1. OpenAPI(Swagger) 및 TypeScript CodeGen 도입
- **정의**: Swagger 문서를 기반으로 프론트엔드 TypeScript 타입을 자동 생성합니다.
- **필요성**: 백엔드 모델 변경 시 프론트엔드 인터페이스를 수동으로 수정하는 반복 작업을 자동화합니다.
- **기대 효과**: 백엔드 수정 시 프론트엔드 빌드 에러가 즉시 발생하여 휴먼 에러 방지.

### 2. Aegis Shield (권한 검사 미들웨어 통합)
- **정의**: 모든 컨트롤러 메서드에 산재한 "권한 확인" 로직을 미들웨어로 통합합니다.
- **필요성**: 새로운 API를 만들 때마다 권한 체크 코드를 넣는 것을 잊어버리는 보안 사고를 원천 방지합니다.
- **기대 효과**: 보안 신뢰도 대폭 향상 및 핵심 로직 집중 가능.

### 3. SignalR 이벤트 상수 공유
- **정의**: 백엔드와 프론트엔드 간 SignalR 이벤트명을 공유 상수로 관리합니다.
- **기대 효과**: 실시간 통신 계층의 최종적인 타입 안정성 완성.
