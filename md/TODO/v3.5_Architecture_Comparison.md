# MooldangBot v3.5.x 아키텍처 결함 및 안정화 비교 분석 보고서

이 문서는 최근 발생한 시스템 가동 중단(빌드 오류 및 런타임 예외)의 원인을 분석하고, 현재 롤백된 **'안정 상태'**와 에러가 발생했던 **'과도기적 상태'**를 기술적으로 비교합니다.

---

## 1. 🔍 핵심 장애 원인 분석 (Root Cause Analysis)

장애의 근본 원인은 로직의 설계 미스가 아닌, **대규모 리팩토링 과정에서의 '참조 파손'과 '물리적 결함'**에 있었습니다.

1.  **레거시의 불완전한 파기**: [ChannelManager](file:///c:/webapi/MooldangAPI/MooldangBot.Presentation/Security/ChannelManagerAuthorizationHandler.cs#10-81)를 코드에서 삭제했으나, [Program.cs](file:///c:/webapi/MooldangAPI/MooldangBot.Api/Program.cs)의 인증 정책(Authorization Policy)과 DI 등록부에서 이를 여전히 참조하여 빌드가 실패했습니다.
2.  **네임스페이스(Namespace) 누락**: 신규 [ChzzkChatClient](file:///c:/webapi/MooldangAPI/MooldangBot.Infrastructure/ApiClients/Philosophy/ChzzkChatClient.cs#25-368)에서 [StreamerProfile](file:///c:/webapi/MooldangAPI/MooldangBot.Domain/Entities/StreamerProfile.cs#6-91) 등 도메인 엔티티를 사용하면서 `using MooldangBot.Domain.Entities;`를 누락하여 컴파일러가 타입을 식별하지 못했습니다.
3.  **DI 컨테이너 구문 오류**: [DependencyInjection.cs](file:///c:/webapi/MooldangAPI/MooldangBot.Application/DependencyInjection.cs) 리팩토링 중 `return services;` 누락 및 중복 등록으로 인해 인프라 서비스 로드로직이 파손되었습니다.
4.  **하이브리드 아키텍처의 충돌**: [ChzzkChannelWorker](file:///c:/webapi/MooldangAPI/MooldangBot.Application/Workers/ChzzkChannelWorker.cs#15-332)(레거시)와 [ChzzkChatClient](file:///c:/webapi/MooldangAPI/MooldangBot.Infrastructure/ApiClients/Philosophy/ChzzkChatClient.cs#25-368)(피닉스)가 일부 기능을 공유하면서 발생하는 의존성 주입 생명주기(Singleton vs Scoped) 충돌 가능성이 감지되었습니다.

## 3. 🌊 물멍의 기술적 조언

현재의 롤백 상태는 **'완전한 신규 구조'와 '검증된 레거시'가 공존**하는 상태입니다. 이를 통해 서비스 중단은 막았으나, 장기적으로는 다음의 정리가 필요합니다.

### ✅ 지금 해야 할 것 (안정화 전략)
1.  **참조 정리**: [Program.cs](file:///c:/webapi/MooldangAPI/MooldangBot.Api/Program.cs)에서 더 이상 사용하지 않는 [ChannelManager](file:///c:/webapi/MooldangAPI/MooldangBot.Presentation/Security/ChannelManagerAuthorizationHandler.cs#10-81) 관련 정책 수동 주석 처리 (이미 완료).
2.  **네임스페이스 검증**: `using` 구문이 정확히 포함되었는지 IDE(Visual Studio 등)의 '사용하지 않는 using 정리' 기능을 활용해 항시 체크.
3.  **빌드 테스트**: 전체 솔루션 단위가 아닌, 프로젝트 단위(`Domain` -> [Application](file:///c:/webapi/MooldangAPI/MooldangBot.Application/DependencyInjection.cs#24-68) -> [Infrastructure](file:///c:/webapi/MooldangAPI/MooldangBot.Infrastructure/DependencyInjection.cs#16-48))로 하향식 빌드 성공 여부를 순차 확인.

### ⚠️ 주의 사항
- **강제 삭제 금지**: 특정 클래스(예: [ChannelManager](file:///c:/webapi/MooldangAPI/MooldangBot.Presentation/Security/ChannelManagerAuthorizationHandler.cs#10-81))를 삭제할 때는 반드시 `Ctrl + Shift + F`로 전체 솔루션에서 해당 클래스를 참조하는 모든 문자열을 찾아 함께 제거해야 합니다.
- **DB 정합성**: 소스 코드가 요구하는 필드(`IsBotEnabled`, `BotChzzkUid` 등)가 실제 DB 테이블에 존재하는지 마이그레이션 쿼리로 항상 확인해야 합니다.

사용 중인 기술 스택은 C# .NET 10, EF Core (MariaDB), MediatR, SignalR이야.

현재 50~100개 이상의 다중 채널을 모니터링하기 위해 구조를 고도화하는 중인데, 아래 3가지 디테일을 적용하려고 해. 사이드 이펙트가 발생하지 않도록 방어 로직이 포함된 안전한 코드가 필요해.

1. 웹소켓 매니저 (좀비 커넥션 및 중복 수신 방지)

2. SignalR 최적화 (그룹 라우팅 및 클라이언트 누락 방지)
- 문제: Clients.All 대신 그룹별(채널별)로 채팅을 쏘려고 하는데, 프론트엔드 연동 로직이 빠지면 메시지가 누락됨.
- 요구사항: 백엔드 SignalR Hub에서 '특정 채널 그룹(JoinGroup)'을 구독/해지하는 메서드와, 프론트/클라이언트 측에서 연결 직후 해당 방을 구독 요청하는 예시 코드 작성.

3. 안전한 종료 (Graceful Shutdown 및 타임아웃 방지)
- 문제: System.Threading.Channels에 캐싱된 남은 채팅을 종료 직전에 MariaDB에 벌크 인서트 하려는데, 기본 5초 타임아웃 때문에 TaskCanceledException이 발생할 수 있음.
- 요구사항: Program.cs에서 HostOptions.ShutdownTimeout을 30초로 늘리는 설정 방법과, BackgroundService의 StopAsync(또는 ExecuteAsync의 취소 토큰 처리)에서 남은 채널 버퍼를 안전하게 SaveChangesAsync 하는 로직 작성.
