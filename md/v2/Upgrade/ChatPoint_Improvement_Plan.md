## 1. 🔐 API 보안 및 테넌트 격리 강제화 (BDD 2-1 준수)

* **발견된 문제:** 현재 `/api/chatpoint/{chzzkUid}`를 비롯한 컨트롤러의 엔드포인트에 인증 검사가 빠져 있어, URL 조작만으로 타 채널의 설정을 읽거나 수정할 수 있는 심각한 보안 허점이 존재합니다.
* **구조 개선안:** * `ChatPointController`의 모든 엔드포인트에 `[Authorize]` 어트리뷰트를 추가하여 인증을 강제해야 합니다.
  * DB 접근 시 주입된 `IUserSession.ChzzkUid`와 요청 URL의 `{chzzkUid}` 값이 일치하는지 확인하고, 불일치할 경우 즉시 `403 Forbidden`을 반환하여 타 테넌트 침범을 원천 차단해야 합니다.

## 2. 🛡️ 데이터 동시성(Concurrency) 경합 방어 (BDD 3-2 준수)

* **발견된 문제:** `ViewerPointEventHandler`가 MediatR을 통해 병렬 실행되므로, 동일 시청자가 매우 빠른 속도로 연속 채팅을 치거나 룰렛을 돌릴 경우 `ViewerProfile`에 대한 DB Update 경합이 발생할 수 있습니다.
* **구조 개선안:** * `ViewerProfile` 엔터티 내부 포인트 및 출석 관련 컬럼(혹은 전용 타임스탬프)에 EF Core의 `[ConcurrencyCheck]` 속성을 부여해야 합니다.
  * 데이터 저장 시 `SaveChangesAsync()`에서 `DbUpdateConcurrencyException` 예외가 발생할 경우, Catch 블록에서 DB의 최신 값을 Reload한 뒤 더할 포인트/카운트를 재계산하여 최대 3회까지 저장을 재시도하도록 로직을 보완해야 합니다.

## 3. 🚦 외부 API 속도 제한 (Rate Limit) 방어 (BDD 3-4 준수)

* **발견된 문제:** 사용자가 `!포인트` 명령어를 쳤을 때 봇 응답에 `{팔로우일수}` 변수가 포함되어 있다면, 매 요청마다 치지직 OpenAPI를 실시간으로 호출하므로 429 Rate Limit 에러를 유발할 위험이 큽니다.
* **구조 개선안:** * 외부 연동 병목을 막기 위해 `IMemoryCache`를 활용하여 시청자별 팔로우 일수 데이터를 일정 시간 캐싱하는 레이어를 추가해야 합니다.
  * 혹은 `ViewerProfile` 내부에 팔로우 날짜를 임시로 저장해두고 활용하는 방식으로 외부 API 연속 호출을 방어해야 합니다.
  * 아울러 HTTP 클라이언트 호출 시 소켓 고갈 방지를 위해 수동 인스턴스화(`new HttpClient()`)를 배제하고 반드시 `IHttpClientFactory`를 주입받아 사용해야 합니다.

## 4. 🚫 포인트 부족 시 로직 차단 및 사용자 피드백 (BDD 3-1 준수)

* **발견된 문제:** 현재 포인트 룰렛 실행 시 시청자의 포인트가 룰렛 비용(`CostPerSpin`)보다 적으면 봇이 아무 반응 없이 조용히 실패 처리만 하고 종료됩니다.
* **구조 개선안:** * 룰렛 이벤트 핸들러에서 잔여 포인트가 부족함을 감지하면, `RouletteService`를 호출하거나 DB를 차감하지 않고 로직을 즉시 종료해야 합니다.
  * 동시에 시청자가 혼란을 겪지 않도록 봇이 "포인트가 부족합니다"라는 멘션 채팅을 해당 시청자에게 전송하도록 피드백 흐름을 추가해야 합니다.

## 5. ⏰ KST 시간대 기반 출석 체계 엄격화 (BDD 3-3 준수)

* **발견된 문제:** 출석일 처리 등에 있어 서버의 UTC 시간과 한국 표준시(KST)가 혼용될 경우 연속 출석 계산에 치명적 버그가 발생할 수 있습니다.
* **구조 개선안:** * UTC 기준 시간이 아닌 KST 시간대를 기준으로 날짜 변경선을 체크해야 합니다.
  * 오늘 첫 출석이 인정되는 순간 `ViewerProfile.LastAttendanceAt` 컬럼에 UTC가 아닌 변환된 KST 시간을 명시적으로 할당하여 DB에 저장해야 합니다.

## 👨‍💻 시니어 파트너(물멍)의 아키텍처 확장 제안

현재 포인트의 소비 경로가 룰렛(`RouletteEventHandler`) 단일 지점에 머물러 있습니다. 향후 시스템이 확장될 것을 대비하여, 곡 신청이나 스트리머 후원 아이템 교환 등 여러 기능에서 포인트를 트랜잭션 안전하게 소비할 수 있도록 **`PointTransactionService`** 같은 별도의 도메인 서비스 계층으로 분리하는 리팩토링을 권장합니다.
